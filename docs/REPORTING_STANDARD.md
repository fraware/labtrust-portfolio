# Reporting Standard for Publishable Results

This document defines the minimum reporting requirements for any result table or summary intended for publication (papers, releases, or external comparison). Adherence ensures statistical credibility and reproducibility. **For peer-review submission, these requirements are mandatory;** "optional" or unsourced tables/figures are not acceptable.

## Requirements

### 1. Sample size (seeds)

- **Minimum 20 seeds** for any published result table, unless explicitly justified (e.g. deterministic scenario, resource limit, or documented exception). Eval scripts default to 20 seeds for publishable runs; 10 allowed only for minimal/CI; use 30 for sensitivity or rebuttal. CI overrides with fewer (e.g. 2–5) and must not be cited as the source of draft tables.
- Justification for fewer than 20 seeds must be stated in the run manifest or accompanying text (e.g. "CI smoke: 5 seeds" vs "Table 1: 20 seeds per setting").

### 2. Statistics to report

For scalar metrics (e.g. tasks_completed, latency, throughput):

- **Mean** and **standard deviation** (stdev).
- **p95** and **p99** (95th and 99th percentiles) where applicable (e.g. latency, replay overhead).
- **Confidence interval** for the mean: 95% CI (e.g. t-interval) when n >= 2; report as `[lower, upper]` or equivalent in the summary JSON.

### 3. Run manifest

Every published result set MUST be accompanied by a **run manifest** that lists:

- **Seeds** used (e.g. `[1, 2, ..., 10]` or the actual list).
- **Scenario(s)** (scenario_id or list).
- **Fault settings** (e.g. drop_completion_prob, delay_fault_prob, or fault_sweep labels).
- **Script and version** (or commit) that produced the run, when feasible.

The manifest may be embedded in the summary JSON (e.g. `run_manifest` in `e3_summary.json`, `multi_sweep.json`) or published as a separate file (e.g. `run_manifest.json` next to the summary). When present, **seed_count** (number of seeds) must be recorded so reviewers can see N.

### 4. Tables and figures (peer-review)

- Every table and figure in the draft must be produced by a script that reads from release or run artifacts. The repro block at the top of the draft must list the exact command for each table and figure. "Optional" or "no dedicated script" figures do not satisfy the peer-review bar; add a plot or export script (with JSON fallback if needed) so the figure is autogenerated.

### 5. Comparison results (A vs B)

When two conditions are compared (e.g. fixed vs meta, robust vs naive, baseline vs feature/regression), the summary JSON SHOULD include under `excellence_metrics` (or equivalent):

- **difference_mean**: raw difference of means (e.g. mean(A) - mean(B)).
- **difference_ci95**: bootstrap or analytical 95% CI for the difference `[lower, upper]`.
- **difference_ci_width** (optional): width (upper − lower) of the difference CI for precision reporting.
- **paired_t_p_value** (optional): two-tailed p-value from paired t-test when n >= 2.
- **power_post_hoc** (optional): approximate post-hoc power of the paired t-test at observed effect size and n.
- **alpha**: significance level (e.g. 0.05).

For publishable comparisons, report either post-hoc power (when p > alpha) or note that the CI excludes zero; optionally report CI width. Scripts `meta_eval.py`, `rep_cps_eval.py`, and `scaling_heldout_eval.py` populate these via `labtrust_portfolio.stats` (paired_t_test, bootstrap_ci_difference, effect_size_mean_diff, power_paired_t_test). See STANDARDS_OF_EXCELLENCE.md.

**P8 meta_eval (paired binary collapse):** Report `collapse_paired_analysis` (Wilson 95% CIs on marginal collapse rates; exact two-sided McNemar on discordant pairs) and distinguish **non-inferior counts** (`meta_non_worse_collapse`, alias `meta_reduces_collapse`) from **strict improvement** (`meta_strictly_reduces_collapse`). Per-arm mean `tasks_completed` CIs use Student’s t (`tasks_completed_ci95_method`); paired continuous summary uses bootstrap on paired resampling (`difference_ci95_method`). When using `--non-vacuous`, record `run_manifest.stress_selection_policy` for pre-specified stress calibration.

### 6. Success / kill criteria (recommended)

Summary JSON may include a `success_criteria_met` or `kill_criteria` block so that paper claims (e.g. "beat baseline", "no safety regression") are machine-checkable. For conditional papers (P2, P5, P6, P8), `success_criteria_met.trigger_met` indicates whether the trigger proof required by `docs/CONDITIONAL_TRIGGERS.md` is satisfied. Export scripts and integration tests can assert on these fields.

## CI vs publishable

- **CI** runs use **reduced seeds** (e.g. 2–5) for speed and smoke. Result tables from CI are not publishable as-is.
- **Publishable** tables and figures must be regenerated with the script’s publishable default (**20 seeds**). The run manifest must state the seed count (e.g. seeds 1..20); document "publishable: 20 seeds" in the runbook or manifest when producing tables for a draft.
- **Script defaults:** Eval scripts default to 20 seeds where applicable (e.g. `maestro_fault_sweep`, `maestro_baselines`, `meta_eval`, `meta_collapse_sweep`, `generate_multiscenario_runs`, `replay_link_e3`, `produce_p0_e3_release`). CI overrides with fewer (e.g. `--seeds 5`, `--seeds 2`). Use `--seeds 10` for quick local runs.
- **Fixed RNG seed (reproducibility):** Set **LABTRUST_FIXED_SEED** (integer) so stochastic steps (e.g. bootstrap in stats) use a fixed seed; run_manifest still records the seeds used per run. Same seed list + same script + same LABTRUST_FIXED_SEED yields reproducible comparison stats. Document in EVALS_RUNBOOK.
- **Sensitivity:** For sensitivity or rebuttals, re-run with **--seeds 30** (or comma-separated list 1..30) where the script supports it (e.g. `meta_eval`, `rep_cps_eval`, `maestro_fault_sweep`, `generate_multiscenario_runs`).

## Where this is applied

- **P0 E3:** `replay_link_e3.py` writes `e3_summary.json` and `p0_e3_variance.json`; both include run_manifest (seeds, scenario_id). Statistics: mean, stdev, p95, p99, 95% CI. Default `--runs 20` for publishable tables.
- **P4 MAESTRO fault sweep:** `maestro_fault_sweep.py` writes `multi_sweep.json` with run_manifest (seeds, scenarios, fault settings). Statistics: mean, stdev, p95, p99 per metric. Default `--seeds 20` for publishable tables; CI may use fewer for speed.
- **P3 Replay:** `replay_eval.py` writes `replay_eval/summary.json` (`schema_version: p3_replay_eval_v0.2`) with replay_level, nondeterminism_budget, divergence_localization_confidence, corpus_outcome_wilson_ci95; `corpus_space_summary`; `per_trace[]` entries including **`corpus_category`** (synthetic_trap, field_proxy, real_ingest, synthetic_pass), trace_json_bytes, state_hash_after_count, diagnostic_payload_bytes_approx, localization_matches_expected; overhead_stats (mean, stdev, empirical p95/p99, CIs; `percentile_method`); baseline_overhead (apply-only, final-hash-only, witness_window ablation, paired full_vs_apply_only); multi_seed_overhead; root_cause_category and witness_slice per divergence; top-level witness_slices; with `--overhead-curve`, overhead_curve[] (event_count, p95_replay_ms, optional p95 CI); with `--l1-twin`, l1_twin_ok, l1_twin_final_hash_match, l1_twin_replay_time_ms, **`l1_twin_summary`** (multi-seed aggregate: n_seeds, all_pass, mean_time_ms, stdev_time_ms, min/max_time_ms); optional process_peak_rss_bytes. Corpus tables: `scripts/export_replay_corpus_table.py` (Table 1 + Table 1b; `--out-md papers/P3_Replay/generated_tables.md` for the paper snapshot). Verify: `scripts/verify_p3_replay_summary.py` (expects corpus_space_summary, corpus_category, and space fields on corpus per_trace rows).
- **Other evals:** P1, P2, P5, P6, P7, P8 summary outputs include run_manifest (seeds, scenario, fault settings, script) and, where stochastic, mean/stdev and 95% CI when n >= 2. **P7** additionally ships `negative_results.json` (discrimination metrics, per-mode latency, lift vs reviewer baselines) and exported `p7_*.csv` tables for the manuscript; see [EVALS_RUNBOOK.md](EVALS_RUNBOOK.md) § P7. P2 additionally reports per-scenario blocks, scheduling-dependent slices, threat micro-evidence, and comparator metadata (`offline_comparator_baselines`) so manuscript claims can be mapped directly to exported tables/figures. See runbook "CI vs publishable" and per-script help.

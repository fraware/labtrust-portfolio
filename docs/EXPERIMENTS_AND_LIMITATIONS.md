# Experiments and Limitations

This document summarizes what the portfolio evaluation does (strengthened experiments) and what it does not do (intentional scope limits). See [PLAN_STRENGTHEN_EXPERIMENTS.md](PLAN_STRENGTHEN_EXPERIMENTS.md) for the full improvement plan and [REPORTING_STANDARD.md](REPORTING_STANDARD.md) for publishable result requirements.

## What we do

- **Sample size and run manifest:** Minimum 20 seeds for publishable tables (10 allowed for minimal/CI); run manifest in each summary JSON records seeds, seed_count, scenario(s), fault settings, and script. Use 30 for sensitivity. See [REPORTING_STANDARD.md](REPORTING_STANDARD.md).
- **Statistics:** Mean, stdev, 95% CI (t-interval or bootstrap) when n >= 2; p95/p99 where applicable (latency, replay overhead).
- **Comparison stats and power:** When two conditions are compared (fixed vs meta, robust vs naive, baseline vs feature), summary JSONs include excellence_metrics: difference_mean, difference_ci95, difference_ci_width, optional paired_t_p_value, power_post_hoc, alpha. Implemented in meta_eval, rep_cps_eval, scaling_heldout_eval via `labtrust_portfolio.stats`. Optional sensitivity sweep at N=10, 20, 30 via `scripts/sensitivity_seed_sweep.py`.
- **Multi-scenario and fault coverage:** Publishable defaults use multiple scenarios (e.g. toy_lab_v0, lab_profile_v0) and fault sweeps where the paper makes a general claim; P4 supports --all-scenarios and includes RetryHeavy adapter (architecturally distinct from Centralized/Blackboard); P5 uses --fault-mix; P8 supports `--scenario regime_stress_v0|regime_stress_v1`, publishable dual-scenario runs, `--stress-preset` high/very_high, and optional `--fallback-adapter retry_heavy` for two-regime comparison; P7 runs over three assurance profiles (lab, warehouse, medical). P2: default scenarios **toy_lab_v0, lab_profile_v0, rep_cps_scheduling_v0**; delay_fault_prob sweep (default: 0,0.05,0.1,0.2); optional **drop_completion_prob sweep** via `--drop-sweep` (default: 0.02 single value); **per_scenario** summaries (aggregated metrics per scenario_id across all delay/drop combinations); profile ablation (Table 6); latency_cost (Table 5); resilience_envelope (Table 7); safety-gate campaign; **scheduling_dependent_eval** (task-level gate effect on scheduling scenario); **freshness_replay_evidence**, **sybil_vs_spoofing_evidence**, **messaging_sim**, **dynamic_aggregation_series**; **excellence_metrics** with comparison statistics (difference_mean, difference_ci95, paired_t_p_value, power_post_hoc on non-scheduling scenarios); optional multi-step aggregation (Table 4) when --aggregation-steps > 1. Toy/lab runs: scoped adapter parity. Scheduling scenario: gated aggregate can change tasks_completed under spoof stress.
- **Expanded corpora:** P1 contracts: 54+ corpus sequences (tiered benchmark: micro, meso, stress/adversarial including cross-key interleaving, delayed release/reassignment, concurrent controller races) with explicit **labeling / anti-circularity** notes in BENCHMARK_SPEC.v0.1.md; eval emits **named comparator policies** (occ_only, lease_only, lock_only, naive_lww, etc.), `detection_metrics_by_class`, `excellence_metrics` (per_class_ci95, split_brain_detection_advantage), **async stress robustness** (`stress_results.json` with delay/skew/reorder sweeps), and **multi-sequence transport parity** (`transport_parity.json` with parity_confidence). See bench/contracts/README.md, docs/RESULTS_PER_PAPER.md (P1). P3 replay: expanded corpus in bench/replay/corpus (core traps, long-horizon and mixed-failure traps, benign pass, two field-style pass traces); eval `replay_eval.py` reports `schema_version: p3_replay_eval_v0.2`, `corpus_space_summary`, per-trace localization/size fields, baselines, multi-seed overhead, Wilson CIs; L1 stub and L1 twin (--l1-twin); L2 aspirational (REPLAY_LEVELS); tables: `export_replay_corpus_table.py` (Table 1 + Table 1b), `verify_p3_replay_summary.py`; optional `generate_p3_replay_corpus_traces.py` for auxiliary JSON; P6: 15 red-team, 6 confusable deputy, 4 jailbreak-style cases (impl/.../llm_planning.py); validator stack allow_list + safe_args + ponr_gate (+ privilege heuristic); optional benign suite (datasets/p6_benign_suite.json) and adaptive suite (p6_adaptive_suite.json). **Corpus expansion:** Documented protocol for adding contract sequences, replay traps, and red-team/confusable deputy cases; run_manifest records corpus size (corpus_sequence_count, replay_trap_count, red_team_case_count, confusable_deputy_case_count). Optional parameterized generator: `scripts/generate_contract_corpus.py` (N-writer contention). See [CORPUS_EXPANSION.md](CORPUS_EXPANSION.md).
- **Reproducibility:** LABTRUST_FIXED_SEED (env) can be set so stochastic steps (e.g. bootstrap) use a fixed seed; run_manifest still records the seeds used per run. Same seed list + same script yields bit-identical traces where the pipeline is deterministic.
- **Real LLM (optional):** P6 supports optional real-LLM mode for red-team and confusable-deputy prompts. Providers: OpenAI/Anthropic (`--real-llm-provider auto` with OPENAI_API_KEY/ANTHROPIC_API_KEY) and Prime Inference (`--real-llm-provider prime` with PRIME_INTELLECT_API_KEY, fallback PRIME_API_KEY, optional PRIME_TEAM_ID). Synthetic plans remain the default for CI and keyless environments. See Phase 7 in PLAN_STRENGTHEN_EXPERIMENTS.md.
- **Formal verification:** At least two Lean wedges (W1, W2) have machine-checked proofs; W3 evidence-bundle verifier has formal spec and soundness statements (proofs pending). See [formal/lean/README.md](../formal/lean/README.md) and Phase 8 in PLAN_STRENGTHEN_EXPERIMENTS.md.
- **Key results / presentable numbers:** P5, P6, P8 DRAFTs include a Key results subsection with concrete metrics; [RUN_RESULTS_SUMMARY.md](../datasets/runs/RUN_RESULTS_SUMMARY.md) snapshots those three papers; `python scripts/export_key_results_p5_p6_p8.py` exports a markdown Key results block. Other papers (e.g. P3) cite their eval JSON and [RESULTS_PER_PAPER.md](RESULTS_PER_PAPER.md). See [EVALS_RUNBOOK.md](EVALS_RUNBOOK.md) (Camera-ready / key numbers).

## What we do not do (and why)

- **Real hardware:** Physical execution is out of scope. All execution is thin-slice simulation (synthetic traces, MAESTRO adapters, replay from trace). Documented in each paper's Limitations where relevant.
- **Full-system formal verification:** We do not verify the autonomous lab end-to-end or the LLM planner's correctness. We verify selected kernels (evidence bundle, gatekeeper, contract validator) as Lean wedges; see formal/lean/README.md.
- **Unbounded threat model:** Fault sets and red-team cases are fixed corpora; we do not claim coverage of all possible attacks or faults. Threat model and fault set are bounded and stated in the papers.
- **Certification claims:** Standards-mapping (P7) is a translation layer only; no certification claim. Other papers state containment, reproducibility, and evidence quality. **Per-paper limitations:** Scope and overclaim checks are stated in each paper's Limitations section: `papers/P0_MADS-CPS/DRAFT.md` through `papers/P8_MetaCoordination/DRAFT.md` (search for "Limitations").

## Formal verification (what is proved)

- **Implemented wedges:** See [formal/lean/README.md](../formal/lean/README.md) for which wedges are implemented and how to build (`lake build` from `formal/lean/`). **W1 (Gatekeeper):** fail-closed theorems fully proved in `Labtrust/W1Gatekeeper.lean`. **W2 (contract validator):** determinism fully proved in `Labtrust/W2Contract.lean`. **W3 (evidence bundle verifier):** formal spec and soundness lemmas in `Labtrust/W3EvidenceBundle.lean`; proofs are pending (currently use sorry). Portfolio bar is met by W1 and W2; W3 full proofs are optional excellence. Python test harness `tests/test_w3_evidence_bundle.py` aligns impl to W3 spec.
- **Not proved:** End-to-end system correctness; LLM planner output correctness; physical device behavior.
